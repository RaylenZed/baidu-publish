# 百家号多账号自动化软文生成系统 — 完整需求与技术方案

## 一、项目背景

### 1.1 业务场景

用户运营百度百家号账号，计划通过CPS返佣模式推广商品盈利。百家号支持的商品分类有18个：图书教育、家用日常、精品服饰、食品生鲜、数码家电、美妆个护、母婴用品、运动户外、鞋靴箱包、汽车用品、珠宝配饰、宠物用品、鲜花园艺、零食干货、粮油调料、医疗保健、家用器械、中医养生。

### 1.2 运营策略

分两个阶段执行：

**第一阶段（涨粉期）**：每个账号只发纯内容软文，不带任何商品链接、价格信息、店铺名等推广内容。目标是先把粉丝量涨到200以上。这个阶段的文章必须看起来像真人写的经验分享。

**第二阶段（变现期）**：粉丝到200后，开始在软文中嵌入CPS商品推广链接，通过返佣变现。

当前重点是第一阶段。

### 1.3 账号矩阵

用户有5个百家号账号，每个账号绑定1-2个商品分类，专注于该品类领域的内容输出。每个账号有独立的cookie用于后续自动发布。

### 1.4 自动化工具

使用 Dify（一个开源的LLM应用开发平台）构建自动化工作流。Dify支持通过YAML格式的DSL文件导入/导出工作流配置，工作流中可以包含代码节点（Python/JavaScript）、LLM节点、迭代节点等。

---

## 二、核心需求

### 2.1 文章生成需求

1. **每次生成的文章都不一样**：不能每次调用LLM都产出同质化内容，需要通过某种机制确保每篇文章在角度、风格、结构、人设等维度都有差异。
2. **严格的品类约束**：选了"母婴用品"就必须写母婴相关内容，绝对不能跑题写成个人成长、时间管理等无关内容。
3. **低AI味**：文章要读起来像真人写的，不能有明显的AI生成痕迹（如"首先...其次...最后..."的机械化过渡、过于工整的排比句、"作为一个xxx"的套话等）。
4. **包含真实细节**：文章中必须出现至少3个真实存在的品牌名/产品名/专业术语，且必须属于对应品类领域。
5. **符合百家号平台规范**：标题15-30字、正文800-1200字、结尾引导读者互动。

### 2.2 多账号批量需求

1. **一次运行为5个账号各生成1篇文章**。
2. **每个账号绑定1-2个分类**，绑定2个分类的账号每次随机选其中一个来写。
3. **输出中要包含账号的cookie和品类信息**，方便后续自动发布环节使用。
4. **使用Dify的迭代节点**遍历账号列表，每个账号走一遍完整的"生成→润色"流程。
5. **支持定时触发**，通过外部cron调用Dify的workflow API，或者使用Dify自带的定时触发功能。

## 三、解决方案设计

### 3.1 核心思路：随机变量池 + 多维组合

为了让每次生成的文章都不同，设计了一个**多维随机变量池**系统。每次运行时，从以下6个维度各随机抽取1个值进行组合：

| 维度 | 说明 | 数量示例（图书教育） |
|------|------|---------------------|
| 切入角度（angle） | 同一品类下的不同内容方向 | 13种（书单推荐、读后感、阅读方法论、冷门好书发现、经典重读、场景化推荐、书中金句分享、一本书改变了我什么、同主题对比阅读、避坑指南、亲子阅读、书与影视对比、职场成长书单） |
| 人设身份（persona） | 写作者的角色设定 | 7种（通勤看书的上班族、全职妈妈、大三学生、中年书虫、职场新人、退休教师、程序员） |
| 写作风格（style） | 文字的语气和调性 | 8种（轻松日常、走心感悟、干货总结、故事叙述、吐槽、文艺清新、接地气、争议讨论） |
| 文章结构（structure） | 文章的组织形式 | 8种（纯叙述体、小标题分段、问答式、倒叙、对比式、清单体、对话体、日记体） |
| 标题风格（title_style） | 标题的写法 | 6种（疑问式、数字式、故事式、反转式、情感式、实用式） |
| 时间场景（time_hook） | 文章的时间/场景背景 | 12种（春夏秋冬、周末午后、深夜、年初年末、发工资后、搬家、节假日等） |

**角度池和人设池是每个品类专属的**（18个品类各有一套），风格池、结构池、标题风格池、时间场景池是**所有品类共用的**。

以图书教育为例，总组合数为：13 × 7 × 8 × 8 × 6 × 12 = **41,664种组合**，基本不会重复。

### 3.2 工作流节点设计

#### 3.2.1 单品类版本（v3，用于单独测试某个品类）

工作流结构：

```
[开始] → [代码节点：随机变量+拼装Prompt] → [LLM：写手初稿] → [LLM：润色降AI味] → [结束]
```

共5个节点，4条边。

**节点详细说明：**

**节点1 - 开始节点**
- 输入变量1：`category`（下拉选择，18个品类，必填）
- 输入变量2：`topic_keyword`（文本输入，可选，用于指定主题关键词）
- 输入变量3：`product_name`（文本输入，可选，用于指定某个具体产品/书名）

**节点2 - 代码节点（Python）：随机变量 + Prompt生成器**
- 接收开始节点的3个变量
- 使用 `random.seed(int(time.time() * 1000) % 2**31)` 确保每次运行的随机种子不同
- 根据选中的品类，从对应的角度池和人设池中随机抽取
- 从通用的风格池、结构池、标题风格池、时间场景池中随机抽取
- **关键设计：在代码节点内部用Python的f-string直接拼装完整的system prompt、user prompt和润色prompt**
- 输出5个字符串变量：`sys_prompt`、`user_prompt`、`polish_sys`、`combo_id`、`category`

**节点3 - LLM节点：写手初稿**
- System prompt：直接引用代码节点输出的 `sys_prompt`（是一个已经填好所有品类、角度、人设等信息的完整prompt字符串）
- User prompt：直接引用代码节点输出的 `user_prompt`
- Temperature: 0.7

**节点4 - LLM节点：润色降AI味**
- System prompt：直接引用代码节点输出的 `polish_sys`
- User prompt：引用写手初稿LLM节点的输出文本
- Temperature: 0.7
- 此节点额外承担**品类一致性检查**的职责：如果初稿跑题了，润色节点有指令要求完全重写

**节点5 - 结束节点**
- 输出：润色后的文章、combo_id（用于追踪本次的变量组合）、category

#### 3.2.2 多账号批量版本（v4，用于生产环境）

工作流结构：

```
[开始] → [构建账号列表] → [迭代节点：遍历每个账号] → [结束]
                                    │
                         迭代内部（每轮处理1个账号）：
                         [选分类+拼Prompt] → [写手初稿] → [润色降AI味] → [组装输出]
```

共8个节点，7条边。其中4个节点在迭代体内部。

**外部节点：**

**节点A - 开始节点**
- 无输入变量（定时触发或API调用触发）

**节点B - 代码节点：构建账号列表**
- 无输入
- 在代码中硬编码5个账号的配置，每个账号包含：name（账号名称）、cookie（百家号cookie）、categories（绑定的1-2个分类列表）
- 输出：`account_list`，类型为 `array[string]`，每个元素是一个账号配置的JSON字符串
- 用户只需修改这个节点中的accounts列表即可管理账号

**节点C - 迭代节点**
- 输入：节点B的 `account_list`
- 遍历方式：串行（非并行，避免LLM限流）
- 输出：每次迭代的 `output_json`

**节点D - 结束节点**
- 输出：`results`，类型为 `array[string]`，包含5个JSON（每个账号一个）

**迭代内部节点：**

**迭代节点1 - 代码节点：选分类 + 拼Prompt**
- 接收迭代当前项（一个账号的JSON字符串）
- 解析JSON，获取该账号的name、cookie、categories
- 从categories中随机选一个作为本次的品类
- 同v3，从对应品类的角度池/人设池 + 通用风格池/结构池等中随机抽取
- 用f-string拼装完整的sys_prompt、user_prompt、polish_sys
- 输出：sys_prompt、user_prompt、polish_sys、account_info（JSON字符串，包含name/cookie/category）、combo_id

**迭代节点2 - LLM：写手初稿**
- 同v3逻辑，引用迭代节点1的sys_prompt和user_prompt

**迭代节点3 - LLM：润色降AI味**
- 同v3逻辑，引用迭代节点1的polish_sys和迭代节点2的输出

**迭代节点4 - 代码节点：组装输出**
- 接收：account_info（来自迭代节点1）、article（来自迭代节点3的润色结果）、combo_id
- 将它们组装成一个JSON字符串输出
- 输出JSON结构：
  ```json
  {
      "account_name": "账号1-图书",
      "cookie": "实际的cookie值",
      "category": "图书教育",
      "combo_id": "账号1-图书-图书教育-A3P2S1R4",
      "article": "标题：xxx\n\n正文..."
  }
  ```

### 3.3 Prompt设计要点

#### 3.3.1 写手初稿的System Prompt结构

```
# 最高优先级指令
你正在为百家号【{品类名}】品类账号写文章。
文章内容必须100%围绕【{品类名}】展开，这是不可违反的硬性要求。

## 品类约束（极其重要，违反则文章作废）
- 文章的主题、场景、提到的产品/品牌/知识点，都必须属于【{品类名}】领域
- 文章中必须至少出现3个与【{品类名}】直接相关的具体产品名、品牌名或专业术语
- 禁止跑题！不要写成个人成长、时间管理、心灵鸡汤等与品类无关的内容

## 你的身份设定
{从人设池中随机抽取的人设，如"一个养了两只猫的独居女生，猫就是她的命"}

## 本次创作参数
- 内容品类：{品类}
- 切入角度：{从角度池中随机抽取的角度}
- 写作风格：{从风格池中随机抽取的风格}
- 文章结构：{从结构池中随机抽取的结构}
- 标题风格：{从标题风格池中随机抽取的标题风格}
- 时间/场景背景：{从时间场景池中随机抽取的场景}

## 写作要求（10条具体规则）
1. 字数800-1200字
2. 纯内容文章，绝对不带任何商品推广、购买链接、价格信息、店铺名
3. 必须包含具体的、真实存在的产品名/品牌名（且必须是该品类领域的）
4. 像真人分享，有个人经历、具体场景、真实情感
5. 不要出现"作为一个xxx"、"今天给大家分享"、"废话不多说"等套话
6. 可以有口语化表达、省略号、感叹词
7. 开头3秒抓住读者
8. 标题15-30字，有吸引力但不标题党
9. 适当加粗1-2个重点句子
10. 结尾引发互动

## 额外要求
{如果用户指定了关键词或产品名，在此补充}

## 输出格式
标题：xxx
正文内容...
```

#### 3.3.2 润色降AI味的System Prompt结构

```
你是文章润色专家。当前品类：【{品类名}】

## 品类一致性检查（最高优先级）
如果文章与该品类严重不符，必须重写使其围绕该品类展开。

## 润色原则
1. 去除AI痕迹：删掉机械化过渡词、排比句、空话
2. 增加真人感：加入口语化表达（"说真的"、"emmm"、"哈哈"），段落长短有变化
3. 保留核心内容：产品名/品牌名保留
4. 百家号适配：标题15-30字，正文800-1200字，结尾互动，加粗1-2个金句
```

#### 3.3.3 为什么Prompt在代码节点中拼装而不是在LLM节点中用模板变量

这是经过3个版本迭代后得出的关键技术决策。

**v1（最初版本）**：在LLM节点的prompt中使用Dify模板变量 `{{#random-code-node.angle#}}` 引用代码节点的输出。问题是：代码节点使用了自定义ID `random-code-node`，但Dify导入DSL后可能重新分配节点ID，导致模板变量引用失效，LLM收到的是原始占位符而不是实际值。

**v2（加强品类约束）**：在v1基础上增强了prompt的品类约束指令，但变量渲染问题依然存在。LLM收到 `{{#random-code-node.category#}}` 这样的原始文本后，误以为用户没有给出品类信息，于是把prompt指令本身"翻译"成了一篇文章——内容是"你得先告诉我写什么品类"。

**v3（最终方案）**：不再依赖Dify的模板变量渲染。在代码节点中用Python的f-string把所有变量值直接拼接进prompt字符串里。LLM节点的system prompt只是 `{{#1780000000002.sys_prompt#}}` 这一个变量引用（引用代码节点输出的完整prompt字符串）。这样即使Dify对节点ID做了映射，只要代码节点的单个输出变量能传到LLM节点，prompt内容就一定是正确的。

### 3.4 品类跑题问题的三层防御

在v1和v2测试中发现LLM容易跑题（比如选了"母婴用品"却写出一篇时间管理鸡汤文），因此设计了三层防御：

**第一层 - System Prompt开头的最高优先级指令**：在prompt最前面放品类硬约束，明确写出"文章必须100%围绕该品类展开""禁止跑题"，整个system prompt中品类名称出现7次以上。

**第二层 - User Prompt的二次强调**：在user消息中再次点明品类、身份、角度，等于下达任务时又说了一遍具体要求。

**第三层 - 润色节点的品类一致性检查**：润色节点在开始润色之前先判断初稿是否跑题，如果严重偏离品类，直接重写而不是只做润色。这是最后一道兜底。

## 六、Dify技术要点

### 6.1 DSL文件格式

- 格式：YAML
- 版本：0.6.0
- 顶层结构：app（应用信息）、dependencies（依赖）、kind（类型）、version、workflow（工作流定义）
- 节点定义在 `workflow.graph.nodes` 中
- 连线定义在 `workflow.graph.edges` 中
- 每个节点有唯一ID、类型（start/code/llm/iteration/end）、位置信息、数据（配置）

### 6.2 迭代节点要点

- `iterator_selector`：指定要遍历的数组变量
- `output_selector`：指定迭代内部最后一个节点的输出变量
- 迭代内部节点通过 `parentId` 关联到迭代节点
- 迭代内部节点引用当前迭代项：`[迭代节点ID, "item"]`
- 支持串行（`is_parallel: false`）和并行执行

### 6.3 代码节点要点

- 语言：Python 3
- 必须定义 `main()` 函数
- 输入通过 `variables` 配置（指定 `value_selector` 从其他节点获取值）
- 输出通过 `outputs` 配置（定义输出变量名和类型）
- 输出类型支持 string、number、array[string] 等

### 6.4 LLM节点要点

- 通过 `prompt_template` 配置system和user消息
- 通过 `model` 配置模型提供商和模型名称
- 变量引用格式：`{{#节点ID.变量名#}}`
- LLM的文本输出通过 `节点ID.text` 引用

### 6.5 当前使用的模型

- 提供商：langgenius/azure_openai/azure_openai
- 模型：gpt-5.2-chat
- Temperature：0.7
