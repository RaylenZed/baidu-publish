### P1（高风险缺陷/一致性问题）

1. **幂等实现不是原子占位，存在并发竞态**（同 key 并发可能重复创建任务）。
   - 证据：
     - 创建前只做查询：`src/backend/app/services/task_service.py:103-107`
     - 创建后才 `set`，且非 `NX`：`src/backend/app/services/task_service.py:352-357`
   - 与宪法“`SET NX EX`”要求不一致。

2. **批量发布草稿接口失败分支会回滚 `publish_failed` 状态。**
   - 证据：`src/backend/app/api/routes/tasks.py:95-111`
   - 说明：`manual_publish` 内部失败会先改状态再抛异常，但此路由失败分支直接 `rollback`，会把失败状态撤销。

3. **账号导入循环内 `rollback()` 粒度错误，可能影响前序已成功项。**
   - 证据：`src/backend/app/services/account_service.py:353-356`

4. **定时 Misfire 语义仍与“24h 内仅补一次”不严格一致。**
   - 证据：
     - 常规触发会处理所有 `next_fire_at <= now`：`src/backend/app/services/schedule_service.py:262-265`
     - Misfire 依赖 `last_fired_at >= threshold`：`src/backend/app/services/schedule_service.py:289-293`
   - 风险：补偿与常规触发边界不清，且 `last_fired_at` 为 `NULL` 的场景难以严格覆盖“仅补一次”。

5. **通知模块定义了 4 类通知，但调用链未闭环。**
   - 证据：
     - 仅看到方法定义，未被业务路径调用：`src/backend/app/services/notify_service.py:36-118`
     - 全局检索无调用：`rg send_cookie_expired|send_content_warning|send_daily_summary`

6. **并发配置与运行配置未对齐。**
   - 证据：
     - 系统设置支持 `max_concurrent_accounts`：`src/backend/app/schemas/settings.py:27`
     - Worker 启动固定 `--concurrency=3`：`docker-compose.yml:83-86`
   - 风险：界面设置与实际并发不一致。

7. **环境模板与文档仍有关键不一致。**
   - 证据：
     - README WebSocket 路径写 `/api/v1/ws/...`：`README.md:157`
     - 实际后端路径是 `/ws/tasks/{task_id}/logs`：`src/backend/app/main.py:94`
     - README 备份用户 `bjh` 与 compose 用户 `bjh_user` 不一致：`README.md:219` vs `docker-compose.yml:18`
     - 根 `.env.example` 仍是旧模板（无 `JWT_SECRET_KEY`、DB 为 sync URL）：`.env.example:6-15`
     - `src/backend/.env.example` 的 `COOKIE_SECRET_KEY` 示例不是 hex 密钥：`src/backend/.env.example:15`
