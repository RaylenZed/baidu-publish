**R-001 [维度：架构决策充分性]**

问题：PRD 规则 TASK-01 要求"同一账号的任务串行执行"，但 CLAUDE.md 仅在架构决策第 2 条提到"同账号强串行"，没有给出**实现机制**。Celery 默认是并发消费的，串行语义需要显式设计（如 per-account queue、分布式锁、或 Celery `chord`/`chain`）。这是系统最核心的并发控制点，缺乏实现指引会导致开发时各自发挥、引入竞态。

建议：在架构决策中明确串行实现方案，例如：
- 方案 A：每个 account 分配独立 Celery queue（`queue=f"account_{account_id}"`），Worker 以 `concurrency=1` 消费各自队列
- 方案 B：使用 Redis 分布式锁（`account:{id}:task_lock`），Worker 拾取前先 acquire，完成后 release
- 同时说明跨账号并发度 `max_concurrent_accounts` 如何与 Celery Worker 数量/并发配置对齐

严重程度：**高**

**R-002 [维度：目录结构合理性]**

问题：CLAUDE.md 目录结构中缺少以下关键目录/文件：
1. **tests/** — PRD 建议 Pytest + httpx，但目录结构中完全没有测试目录规划
2. **middleware/** — PRD 要求登录限流（C-012）、请求级 `request_id` 注入（日志规范要求）、CORS 配置等，这些是中间件职责，当前无处安放
3. **utils/** 或 **helpers/** — Cookie 加解密、JSONP 回调剥离（PRD PUB-02/PUB-06 提到 `bjhdraft(...)` / `bjhpublish(...)` 格式）、combo_id 格式化等工具函数无归属
4. **v7 核心代码**的位置 — PRD 明确要"复用 v7 已调通的 600+ 行核心 API 集成代码"，但目录结构没有标注 v7 代码在哪里、如何引入

建议：补充以下目录：
```
src/backend/
  app/
    middleware/              # 限流、request_id、CORS
    utils/                   # 通用工具函数
  tests/
    unit/
    integration/
    conftest.py
  v7_legacy/                 # 或在 services/ 中标注 v7 代码迁移位置
```

严重程度：**中**

**R-003 [维度：编码规范完整性]**

问题：CLAUDE.md 缺少**分页规范**。PRD 的任务列表（6.3.4）、文章列表（6.4.1）、系统日志（6.7）等多处需要分页，API 清单中也有 `page` / `size` 参数，但编码规范中没有统一分页的：
- 请求参数命名（`page` vs `offset`、`size` vs `limit` vs `per_page`）
- 响应结构（是否包含 `total`、`has_next`、`total_pages`）
- 默认值与上限（`size` 默认 20、最大 100？）
- 游标分页 vs 偏移分页的选择

建议：在 API 规范中新增分页约定，例如：
```
分页请求：?page=1&size=20（page 从 1 开始，size 默认 20，最大 100）
分页响应：{ "items": [...], "total": 150, "page": 1, "size": 20, "pages": 8 }
```

严重程度：**中**

**R-004 [维度：一致性]**

问题：PRD 定义了**品类枚举**（18 个品类：图书教育、家用日常、精品服饰……），这是系统的核心业务数据，但 CLAUDE.md 没有将品类枚举列为编码常量或枚举定义。这会导致：
- 前后端各自硬编码品类列表，不一致风险高
- 品类→百家号文章分类的映射表（PRD PUB-05）无处定义
- 校验规范 4.5 提到 `categories` 值需在品类枚举内，但枚举本身未给出

建议：在 CLAUDE.md 中新增"业务常量"章节，明确：
1. 18 品类的 Python 枚举类定义位置（如 `models/enums.py` 或 `core/constants.py`）
2. 品类→百家号分类映射表的维护位置
3. 前端同步使用后端 API 获取品类列表，不本地硬编码

严重程度：**中**

**R-005 [维度：编码规范完整性]**

问题：CLAUDE.md 缺少**错误码枚举**和**异常处理分层**规范。当前仅在 API 规范 4.2 第 4 条给了一个错误响应结构示例（`code: "TASK_TIMEOUT"`），但没有：
- 错误码命名规则（`MODULE_ACTION_REASON` 还是其他？）
- 异常在哪一层抛出、哪一层捕获（Service 抛业务异常 → Route 层转 HTTP 响应？）
- 自定义异常基类的设计
- PRD 中的 `error_type` 枚举值（`publish_failed_draft_saved`、`aigc_timeout`、`timeout`、`connection_error`、`parse_error`、`empty_response`）需要统一管理

建议：新增异常处理规范：
1. 定义 `app/core/exceptions.py`，包含 `AppException(code, message, status_code)` 基类
2. 在 `main.py` 注册全局异常处理器，统一转换为 API 规范中的错误响应格式
3. 在 CLAUDE.md 中列出已知 `error_type` 枚举值清单
4. Service 层抛业务异常，Route 层不做 try/except（由全局处理器兜底）

严重程度：**中**

**R-006 [维度：架构决策充分性]**

问题：PRD 规则 TASK-03 规定"单账号每日执行上限默认 3 次"，TASK-07 规定"幂等去重 + 60s 防重"，TASK-08 规定"连续失败 3 次冷却 30 分钟"。这三条规则都需要**状态查询与计数**，但 CLAUDE.md 没有说明这些计数/状态检查的实现位置和时机：
- 每日上限的"一天"按什么时区计算？（Asia/Shanghai？UTC？）
- 计数是在创建任务时查询 DB，还是用 Redis 计数器？
- 冷却 30 分钟的状态存在哪里？如何在 Worker 拾取前检查？

建议：在架构决策中补充：
1. 每日上限按 `Asia/Shanghai` 时区的自然日计算（与调度时区一致），在 `task_service.create_task()` 中查询 `tasks` 表当日该账号记录数
2. 幂等 key 存 Redis（TTL 5 分钟），防重检查在 Service 层 `create_task()` 入口
3. 连续失败冷却：在 `task_service` 中查询该账号最近 3 条任务状态，若连续 failed 则检查最后一条的 `finished_at + 30min > now()`

严重程度：**中**

**R-007 [维度：可操作性]**

问题：CLAUDE.md 目录结构中 `repositories/` 层的职责定义为"数据访问层（复杂查询与事务边界）"，但对于本项目（单用户、低并发、业务逻辑不算复杂），Service 直接使用 SQLAlchemy Session 进行查询已经足够。引入 Repository 层会导致：
- 每个实体多一层样板代码（Repository + Service）
- 简单 CRUD 需要跨两个文件
- 对 AI 编码来说增加了不必要的间接层

建议：考虑两种方案：
- 方案 A（推荐）：**删除 `repositories/` 层**，Service 直接操作 SQLAlchemy Session，复杂查询写为 Service 方法或独立的 query builder
- 方案 B：保留 Repository 层，但明确只有 3+ 行的复杂查询才放入 Repository，简单 CRUD 仍在 Service 中直接操作

严重程度：**低**

**R-008 [维度：技术栈合理性]**

问题：PRD 文章详情页要求"Markdown 渲染预览"和"初稿 vs 润色对比"（左右两栏对照展示），前端还需要"在线 Markdown 编辑器"（6.4.3），但 CLAUDE.md 技术栈中没有提及任何 Markdown 渲染或编辑库。Vue 3 生态中常见选择有：
- 渲染：`markdown-it` 或 `marked`
- 编辑：`md-editor-v3`、`bytemd`、`milkdown`
- Diff 对比：`diff2html` 或自定义

建议：在技术栈 2.2（建议采用）中补充：
```
| Markdown 编辑/渲染 | md-editor-v3 [建议] | Vue3 原生支持，自带预览和工具栏 |
| 文本差异对比 | diff2html [建议] | 用于初稿 vs 润色的 side-by-side diff |
```

严重程度：**低**

**R-009 [维度：一致性]**

问题：PRD 中 `tasks` 表有 `warning` 字段（VARCHAR(30)），`articles` 表有 `content_warning` 字段（VARCHAR(30)），两者都可能存储 `partial_content` 值。CLAUDE.md 架构决策第 10 条仅提到"partial_content 强制降级为 draft_only，并写入 warning"，但没有明确：
- `warning` 写入 `tasks.warning` 还是 `articles.content_warning`，还是两处都写？
- 两个字段的语义区别是什么？（task 级别的警告 vs article 级别的内容警告？）
- 前端展示时读取哪个字段？PRD 6.4.1 文章列表的"警告标签"读的是 article 的 `content_warning`

建议：在架构决策中明确双写策略：
1. `tasks.warning` = 任务执行过程中的警告（AI 影响执行流程的异常状态）
2. `articles.content_warning` = 文章内容层面的警告（展示给用户，影响能否发布）
3. `partial_content` 场景：两处都写入，task 层记录"为什么降级"，article 层记录"内容有什么问题"
4. 前端文章列表读 `articles.content_warning`，任务详情读 `tasks.warning`

严重程度：**低**

**R-010 [维度：编码规范完整性]**

问题：CLAUDE.md 缺少**数据库事务管理**规范。PRD 中有多处需要事务保证的操作：
- 创建任务时同步写入 `tasks` + `content_events`（TASK 创建事件）
- 发布成功时更新 `articles.publish_status` + 写入 `publish_attempts` + 写入 `content_events`
- 删除账号时级联检查未完成任务（ACC-05）
- 定时任务触发时为多个账号批量创建 task

当前没有规范说明：事务在哪一层管理？是 Route 层开事务，还是 Service 层？是否使用 `async with session.begin()` 模式？

建议：新增事务管理规范：
1. 事务边界在 **Service 层**，每个 Service 方法默认一个事务
2. 使用 FastAPI 依赖注入的 `AsyncSession`，Route 层通过 `deps.py` 获取 session
3. 模式：`async with session.begin():` 包裹需要原子性的操作
4. 跨 Service 调用时，传递同一个 session 保证事务一致性

严重程度：**中**

---

**R-011 [维度：架构决策充分性]**

问题：PRD 11.4 中 WebSocket 实时日志接口 `WS /ws/tasks/{id}/logs` 使用 `?ticket=xxx` 鉴权，CLAUDE.md 架构决策第 9 条也提到了 ws_ticket 机制。但缺少以下关键设计：
1. 日志从 Worker 到 WebSocket 的**传输通道**是什么？Worker 和 Backend 是不同进程/容器，日志如何跨进程推送？
2. PRD 提到"任务完成后日志固化在数据库"——那实时推送时日志是从 DB poll 还是从 Redis pub/sub？
3. 多个客户端同时订阅同一个 task 的日志流，如何处理？

建议：在架构决策中补充 WebSocket 日志流的数据通路：
1. Worker 执行每步时写入 `task_logs` 表，同时 publish 到 Redis channel `task:{task_id}:logs`
2. Backend 的 WebSocket handler 订阅对应 Redis channel，实时推送给客户端
3. 客户端连接时，先从 DB 拉取已有日志（回放），再切换为 Redis channel 实时流
4. 技术栈中补充 Redis Pub/Sub 作为日志通道的角色（当前仅提到"Celery Broker + ws_ticket + 日志通道"，但"日志通道"没有具体方案）

严重程度：**高**

---

**R-012 [维度：可操作性]**

问题：CLAUDE.md 的开发进度表（第 6 节）列出了 22 个模块，全部"未开始"。但没有给出**模块间的依赖关系**和**建议开发顺序**。PRD 第六部分有 Phase 1-4 的分期计划，但 CLAUDE.md 没有将其映射为具体的模块依赖图。对于 AI 或开发者来说，缺乏依赖关系会导致：
- 不知道先做哪个模块
- 可能先做了依赖后续模块的功能
- 无法判断某个模块是否可以独立开发和测试

建议：在进度表中增加依赖标注，或新增"模块依赖图"章节，例如：
```
基础设施 → 数据库 Schema → 认证模块 → 账号管理
                                    ↘ 任务编排 → Celery Worker → 任务面板
                                                              → WebSocket 日志
                                    ↘ 变量池 → 任务编排（依赖变量池生成 combo）
```

严重程度：**低**

## 评审回复（架构方）

**R-001 回复**：同意。将明确采用“Redis 分布式账号锁 + Celery 并发对齐”方案：Worker 执行前先获取 `account:{account_id}:run_lock`（SET NX EX），未获取则延迟重试；`max_concurrent_accounts` 作为全局并发上限，与 Worker 并发参数保持一致，并在系统设置变更时同步生效策略（重启 Worker 或动态下发）。

**R-002 回复**：部分同意。同意补充 `tests/`、`middleware/`、`utils/` 的落位与职责；关于 `v7_legacy/`，不单独长期保留一套平行代码目录，采用“迁移到 `services/` + 必要时在 `integrations/` 放兼容适配层”的方式，避免双份逻辑长期分叉。

**R-003 回复**：同意。补充统一分页规范：请求固定 `page`、`size`（`page` 从 1 开始，`size` 默认 20、最大 100）；响应固定 `{items,total,page,size,pages}`；本项目列表接口默认偏移分页，超大数据量场景再评估游标分页。

**R-004 回复**：同意。补充“业务常量”规范：18 品类作为后端单一枚举源（如 `core/constants.py` / `models/enums.py`）；维护“品类 -> 百家号分类”映射表；前端通过 API 拉取枚举，不允许本地硬编码复制一份。

**R-005 回复**：同意。补充异常分层与错误码规范：新增 `core/exceptions.py`（`AppException` 基类及子类）；`main.py` 注册全局异常处理器统一输出错误结构；Service 抛业务异常、Route 不做分散 try/except；统一维护 `error_type` 枚举与错误码命名约定。

**R-006 回复**：同意。补充规则落地时机：创建任务入口统一执行“每日上限/幂等/60s 防重/冷却”检查；自然日按 `Asia/Shanghai` 计算；幂等 key 存 Redis（TTL 5 分钟）并保留 DB 兜底校验；冷却规则按账号最近失败链路判定并在创建前与 Worker 执行前双重校验。

**R-007 回复**：部分同意。保留 Repository 但降级为“按需使用”：简单 CRUD 允许 Service 直接操作 Session；仅复杂查询、跨表聚合、可复用事务片段进入 Repository，避免样板层级膨胀。

**R-008 回复**：同意。补充前端文档技术建议：Markdown 编辑/渲染采用 `md-editor-v3`，对比视图采用 `diff2html`（或等价方案），满足“在线编辑 + 预览 + 初稿/润色对照”。

**R-009 回复**：同意。明确双字段语义与写入策略：`tasks.warning` 表示任务执行级警告，`articles.content_warning` 表示内容级警告；`partial_content` 场景双写，文章列表读取 `content_warning`，任务详情读取 `warning`。

**R-010 回复**：同意。补充事务规范：事务边界在 Service 层；使用同一 `AsyncSession` 并通过 `async with session.begin()` 包裹原子操作；跨 Service 调用传递同一 session；涉及状态更新+审计写入的场景强制单事务提交。

**R-011 回复**：同意。补充 WebSocket 日志链路：Worker 写 `task_logs` 后同步发布 Redis Pub/Sub 频道 `task:{task_id}:logs`；WebSocket 端先回放 DB 历史日志，再订阅频道推送增量；支持同任务多客户端订阅，ticket 仅用于连接鉴权不影响广播。

**R-012 回复**：同意。补充”模块依赖与建议顺序”章节，将 PRD Phase 1-4 映射到模块依赖链，标注每个模块的前置条件与可并行开发模块，避免无序启动造成返工。
