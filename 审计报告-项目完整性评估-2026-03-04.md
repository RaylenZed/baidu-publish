# 百家号自动化发布系统审计报告（完整性评估）

- 审计日期：2026-03-04
- 审计范围：
  - `PRD-百家号自动化内容发布管理系统.md`
  - `CLAUDE.md`（项目宪法）
  - `开发路线图.md`
  - `src/backend`、`src/frontend`、`docker-compose.yml`、`README.md`
- 审计方式：文档对照 + 代码静态核查 + 本地可运行性验证（编译/构建/测试命令）

## 1. 审计结论（Executive Summary）

项目当前**未达到“完整开发完毕”**标准，暂不具备上线条件。

- 综合判定：`部分完成（后端主链路可见） + 关键交付项缺失（迁移/前端构建/测试/规则闭环）`
- 上线判定：`不通过`
- 风险等级：`高`

## 2. 阻断上线项（P0）

1. 数据库迁移脚本缺失（只有 Alembic 框架，无版本文件），容器启动依赖 `alembic upgrade head`，会导致部署不可控。
2. 前端构建失败（`npm run build` 报 TS 错误），无法稳定产出 `dist`。
3. 定时任务 Cron 校验实现错误（`croniter(..., retime=True)` 参数无效），调度模块存在直接不可用风险。
4. 登录 token 字段前后端不一致（前端读 `token`，后端返回 `access_token`），登录后鉴权链路会中断。

## 3. 高优先级问题（P1）

1. `POST /tasks/publish-drafts` 仍为占位实现，未接文章发布服务。
2. `retry_of_task_id` 语义未真正落库到重试任务。
3. 任务关键规则与宪法/PRD偏差：
   - 日限按 UTC 计算，未按 `Asia/Shanghai` 自然日。
   - 冷却逻辑是“窗口内失败计数”，非“最近3条连续失败”。
   - 创建任务 `category` 被强制必填，未支持 PRD 约定“留空随机”。
4. Beat 计划不完整：仅注册超时回收；`fire_schedules`、`recover_misfire` 未纳入 `beat_schedule`。
5. 通知/审计未闭环：
   - `task_created` 未实际写入审计。
   - `daily_summary`、`content_warning`、`cookie_expired` 仅定义未形成完整触发链。
6. 前后端数据契约不一致（例如账号列表分页结构预期不一致），存在运行时错误风险。
7. 系统日志前端页缺失（仅有后端接口），与宪法目录结构不一致。
8. 前端硬编码品类，违反“前端通过 API 拉取品类列表”的宪法要求。

## 4. 可运行性验证结果

### 4.1 后端

- `python3 -m compileall -q src/backend/app`：通过
- `cd src/backend && pytest -q`：`no tests ran in 0.01s`

结论：语法层面可导入，但缺乏有效自动化测试覆盖，无法证明“功能完成+回归安全”。

### 4.2 前端

- `cd src/frontend && npm install`：通过
- `cd src/frontend && npm run build`：失败（13 个 TS 错误）

结论：前端当前不是可交付状态。

## 7. 整改优先级建议

3. 修正 `auth` token 字段契约（前后端统一为 `access_token`）。
4. 修复 `schedule_service` 的 cron 校验实现，并补齐 beat 定时注册。

1. 增加 `tests/conftest.py`、单元测试、集成测试，至少覆盖核心成功/失败路径。

3. 清理 README 与 `.env` 指南，统一部署入口与变量口径。

## 8. 最终结论

本项目已经有较完整的后端骨架和业务主流程雏形，但从“完整开发完毕”与“可上线交付”标准看，当前仍处于**未收口状态**。  
建议按本报告“第一批/第二批/第三批”顺序推进，完成后再做一次上线前复审。
